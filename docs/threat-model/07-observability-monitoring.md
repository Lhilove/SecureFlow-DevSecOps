# Threat Model — Observability & Monitoring

## System Overview

The observability stack provides visibility into the health, security, and performance of the SecureFlow platform. It consists of Prometheus for metrics collection, Grafana for visualization and dashboards, Loki for centralized log aggregation, and Alertmanager for security and operational alerts.

Application logs, Kubernetes events, and infrastructure metrics are continuously collected to support incident detection, troubleshooting, and forensic investigations. Because these systems contain sensitive operational information and support incident response, maintaining their integrity and availability is critical.

## STRIDE Analysis

### Spoofing

- Attacker impersonates a monitoring component and submits fake metrics.
- Unauthorized user logs into Grafana using stolen credentials.
- Compromised application sends forged logs to Loki.

### Tampering

- Security alerts modified or deleted.
- Grafana dashboards altered to hide suspicious activity.
- Logs modified to remove evidence of an attack.
- Alerting rules disabled or weakened.
- Prometheus scrape configurations changed to exclude critical workloads.

### Repudiation

- Attackers delete audit logs after compromising the environment.
- Administrative actions are not recorded.
- Missing immutable logging prevents attribution.
- Dashboard modifications occur without audit trails.

### Information Disclosure

- Grafana dashboards expose sensitive infrastructure information.
- Application logs contain passwords, API keys, or JWT tokens.
- Prometheus metrics expose internal IP addresses or service names.
- Loki stores personally identifiable information (PII).
- Monitoring endpoints are publicly accessible.

### Denial of Service

- Excessive metrics overwhelm Prometheus.
- Log flooding exhausts Loki storage.
- Alert storms overwhelm incident responders.
- Large dashboard queries consume Grafana resources.
- Disk exhaustion causes loss of monitoring data.

### Elevation of Privilege

- Overly permissive Grafana roles grant administrative access.
- Attackers obtain credentials stored within monitoring systems.
- Compromised monitoring service accesses Kubernetes resources beyond its intended permissions.
- Service accounts have unnecessary cluster-wide privileges.

## Attack Scenarios

### Scenario 1: Log Tampering (Critical)

An attacker compromises an application pod and gains access to the logging infrastructure. They delete or overwrite security logs before incident responders detect malicious activity.

Impact: Critical; forensic evidence is lost, delaying incident response and making post-incident investigations significantly more difficult.

---

### Scenario 2: Alert Suppression (High)

An attacker gains administrative access to Grafana or Alertmanager and disables alerts related to suspicious authentication attempts and runtime anomalies.

Impact: High; attacks continue undetected, increasing the likelihood of successful compromise and data theft.

---

### Scenario 3: Sensitive Data Exposure Through Logs (High)

Application developers accidentally log JWT tokens, database credentials, or customer information. Attackers with access to Loki retrieve these logs and use the exposed information to compromise other systems.

Impact: High; credential exposure, customer privacy violations, and regulatory compliance failures.

---

## Mitigations

### Existing Controls

- Centralized logging through Loki.
- Metrics collected using Prometheus.
- Dashboards provided by Grafana.
- Runtime events generated by Falco.
- Kubernetes audit logs support incident investigations.

### Gaps and Recommendations

- Enforce Role-Based Access Control (RBAC) within Grafana.
- Encrypt logs both in transit and at rest.
- Prevent sensitive data from being written to logs.
- Enable immutable log storage and retention policies.
- Monitor for unexpected changes to alerting rules.
- Restrict access to Prometheus, Loki, and Grafana.
- Implement multi-factor authentication for monitoring administrators.
- Regularly review alert configurations.
- Enable audit logging for Grafana administrative actions.
- Continuously monitor log ingestion rates to detect log flooding attacks.
